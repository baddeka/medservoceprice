# MedServicePrice.kz

**Агрегатор и сравнение цен на медицинские услуги в Казахстане** — анализы, приёмы
врачей, диагностика. Как Aviasales, но для медицины. MVP для хакатона 2025 (Кейс 1).

Пациент сегодня вынужден вручную обходить десятки сайтов клиник, чтобы сравнить цену
анализа крови или приёма врача. MedServicePrice собирает публичные прайсы из открытых
источников, нормализует разнородные названия к единому справочнику и даёт быстрый поиск
с фильтрами и сравнением цен.

---

## Архитектура (MVP-версия)

Сбор данных — отдельный изолированный пайплайн (не сервис), API читает только готовую БД:

```
  parse.py  ──►  raw_listings  ──►  normalize.py  ──►  listings + service_dictionary  ──►  FastAPI API  ──►  Frontend
 (сбор сырья)     (raw-слой)       (fuzzy-match,        (чистые данные)                   (/search …)      (поиск, фильтры,
                                    дедуп, справочник)                                                       сравнение)
```

| Слой | Технология | Файлы |
|------|-----------|-------|
| Парсинг | Python + requests/BeautifulSoup | `parse.py` |
| Нормализация | Python + rapidfuzz (fallback: difflib) | `normalize.py`, `service_dictionary.json` |
| База данных | SQLite (один файл `medprice.db`) | `backend/db.py` |
| API | FastAPI | `backend/main.py` |
| Frontend | Vanilla JS + HTML/CSS (без сборки) | `frontend/` |

**Почему SQLite, а не Postgres, и почему ручной запуск, а не scheduler** — осознанное
решение под формат «24 часа, 2 человека». Архитектура (raw-слой, дедупликация,
нормализация, эндпоинт `/admin/trigger-parse`) спроектирована так, что переход на cron и
Postgres — это замена строки конфигурации, а не переписывание ядра (см. «План развития»).

## Быстрый старт

Требуется Python 3.10+.

```bash
# 1. Установить зависимости
pip install -r requirements.txt

# 2. Наполнить базу демо-данными (чтобы приложение работало сразу, офлайн)
python seed_demo.py

# 3. (опционально) Собрать реальные данные с сайтов клиник
python parse.py        # сбор сырья -> raw_listings
python normalize.py    # нормализация -> listings + справочник

# 4. Запустить сервер (отдаёт и API, и фронтенд)
uvicorn backend.main:app --reload
```

Открыть **http://localhost:8000** — фронтенд раздаётся тем же сервером.
Документация API (Swagger): **http://localhost:8000/docs**.

> Запускать из корня проекта (папка `MedServicePrice`). Пути к `medprice.db` вычисляются
> относительно файлов, так что БД у всех скриптов одна и та же.

## API (контракт)

| Метод | Путь | Назначение |
|-------|------|-----------|
| `GET` | `/search` | Поиск + фильтры + сортировка |
| `GET` | `/services/autocomplete` | Автодополнение по справочнику |
| `POST`| `/admin/trigger-parse` | Ручной запуск парсинга «через интерфейс» (ТЗ п.3.1) |
| `GET` | `/cities`, `/categories`, `/stats`, `/clinic/{name}`, `/health` | Вспомогательные |

**`GET /search`** — параметры: `q`, `city`, `category`, `price_min`, `price_max`,
`sort` (`price_asc`/`price_desc`/`date_desc`), `include_stale`.
Каждый элемент ответа: `{clinic_name, service_name, category, price_kzt, city, parsed_at,
source_url, …}`. Дополнительно отдаются `is_cheapest`, `is_stale`, `updated_days_ago`,
адрес/телефон/часы работы — для карточек.

**`POST /admin/trigger-parse`** → `{status, new_records, errors}`. Синхронно запускает
`parse.py` → `normalize.py` (в подпроцессах, падение источника не роняет API). Это та же
кнопка «Обновить данные» в интерфейсе. Превращается в cron одной строкой:

```cron
0 3 * * * curl -X POST http://localhost:8000/admin/trigger-parse
```

## Как покрыты требования ТЗ

| Требование ТЗ | Где реализовано |
|---------------|-----------------|
| Сбор с открытых источников, raw-слой отдельно | `parse.py` → `raw_listings` |
| Дедупликация при повторном запуске | `price_hash` (UNIQUE) в `listings` |
| Журналирование ошибок парсинга | `parse_errors.log` + видны в ответе `/admin/trigger-parse` |
| Запуск парсинга вручную через интерфейс / cron | `POST /admin/trigger-parse` + кнопка на фронте |
| Нормализация к единому справочнику (≥50 позиций) | `normalize.py` + `service_dictionary.json` (**73 позиции**) |
| Очередь ручной разметки (unmatched) | несматченные → `unmatched.csv` |
| Поиск с автодополнением | `/services/autocomplete` + дропдаун на фронте |
| Фильтры: город, категория, ценовой диапазон | панель фильтров + `/search` |
| Сортировка: по цене, по дате обновления | `sort=` в `/search` |
| Карточка клиники (все услуги) | `/clinic/{name}` + модалка |
| Дата последнего обновления цены | поле `parsed_at`, метка «обновлено N дн. назад» |
| Не выдавать данные старше 30 дней как актуальные | по умолчанию `/search` прячет, метка «устарело» |
| Адаптивный интерфейс (мобильные) | CSS-сетка с брейкпоинтами 860/560px |
| Отказоустойчивость парсера | try/except по каждому источнику; подпроцессы в trigger-parse |
| Сравнение нескольких клиник (доп. функция) | чекбоксы «сравнить» + таблица сравнения |

## План развития (что вынесено за рамки 24-часового MVP)

- **Scheduler (cron/APScheduler)** — уже есть точка входа `/admin/trigger-parse`, нужен
  лишь cron-триггер.
- **Карта клиник (Leaflet/2GIS)** — координаты добавляются в `listings`, фронту нужен слой карты.
- **История изменения цен** — структура (`price_hash`, `parsed_at`) уже это позволяет;
  достаточно не перезаписывать, а накапливать срезы.
- **AI-нормализация unmatched-очереди** — сейчас простой fuzzy-match; батч через LLM
  поднимет покрытие на длинном хвосте названий.
- **Подписка на изменение цены**, переход на **PostgreSQL** (меняется connection string).

## Команда и разделение работ

- **Дорожка 1 (данные):** `parse.py`, `normalize.py`, справочник синонимов.
- **Дорожка 2 (продукт):** `backend/`, `frontend/`, `seed_demo.py`.

Контракт API и схема БД зафиксированы заранее, поэтому дорожки разрабатывались
параллельно и сходятся только в двух точках синхронизации.

> Примечание: `rapidfuzz` даёт лучшее качество fuzzy-match, но если колесо под вашу
> версию Python не собирается, `normalize.py` автоматически использует встроенный
> `difflib` — приложение работает без него.
